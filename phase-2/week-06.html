Week 6 — Analyst-Grade SQL

Phase 2 · SQL Skill Focus: Iterate with AI to Refine Cert: Kaggle Advanced SQL ~12 hours
Course Progress
Week 6 of 12

The SQL you write this week is what a senior analyst writes on an average Tuesday. Window functions, date arithmetic, case statements, top-N-per-group. Interview whiteboard problems often hinge on these. By week's end you've earned the Kaggle Advanced SQL cert and you're halfway through the course.

Learning Goals

  • Master window functions: ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, running aggregates
  • Handle dates properly: period-over-period, rolling windows, date truncation
  • Write CASE statements cleanly to turn raw data into categorical insight
  • Drill Skill #4: iterate with AI to refine — building complex queries step by step
  • Earn Kaggle Advanced SQL cert
  • Ship a period-over-period analysis that would survive a real analyst code review

Why Window Functions Matter

Questions like “what was each customer's rank in transaction volume within their segment” or “what was the month-over-month change in revenue” are hard to solve with GROUP BY alone. You need window functions.

Before window functions, analysts solved these problems with self-joins or correlated subqueries. The queries were slow, unreadable, and error-prone. Window functions made them fast and clean. That's why they show up in interviews. They're the test of whether you know modern SQL.

AI Co-Work Pattern — Skill #4: Iterate with AI

The iteration drill

Window functions are where you can't just prompt once and trust the answer. The first query Claude writes will often be syntactically valid but semantically wrong. You iterate:

  1. Ask Claude for a first version
  2. Run it in DB Fiddle against a small sample (LIMIT 10)
  3. Check the result manually against what you expected
  4. When it's off, tell Claude specifically what's off. Not “this is wrong” — say “I expected customer X to be rank 1 in segment Y but they're showing as rank 3. Did you partition by the right thing?”
  5. Claude revises. You test again. Loop.

Three or four iterations is normal for a complex window query. That's not Claude being bad. That's how analysts actually work.

The Five Drills

Drill 1 — Ranking window functions (ROW_NUMBER, RANK, DENSE_RANK)

Explain the three ranking window functions. Specifically:

1. What's the difference between ROW_NUMBER, RANK, and DENSE_RANK
   when there are ties?
2. Write a query that returns the top 3 customers by total transaction
   amount within each segment (top-3-per-group, the classic)
3. What happens if two customers tie for 3rd place? Show me the
   output with each of the three functions and explain what you'd
   pick and why

Run COUNT(*) checks against the full result so I can verify the
row count is what I'd expect (number of segments times 3, give or
take ties).

Drill 2 — LAG and LEAD (period-over-period)

LAG and LEAD let you reference previous or next rows within a
window. This is how you do month-over-month change. Walk me through:

1. Syntax: LAG(column, offset) OVER (PARTITION BY ... ORDER BY ...)
2. Write a query: for each month in 2025, show total transaction
   amount, prior month's amount, and the delta (this month minus
   last)
3. Write a variant: the percent change month-over-month
4. Explain what happens for the first month (LAG returns NULL) and
   how to handle that

Drill 3 — Running aggregates (running totals, moving averages)

Window functions can also do running totals and moving averages.
This is the framed window: ROWS BETWEEN ... PRECEDING AND CURRENT
ROW.

1. Write a query for cumulative transaction amount per account,
   ordered by transaction_date (classic running total)
2. Write a query for 7-day moving average of transaction amount
   per account
3. Write a query for 30-day trailing sum of deposits per customer

Explain the difference between ROWS BETWEEN and RANGE BETWEEN and
when you'd use each.

Drill 4 — Date arithmetic and truncation

Date handling is where interview candidates crash. Walk me through:

1. DATE_TRUNC (PostgreSQL syntax) for rolling up transactions to
   month, quarter, year
2. EXTRACT for pulling year, month, day-of-week out of a date
3. Date arithmetic: how many days between two dates, adding/
   subtracting intervals
4. The common gotcha: timezone handling when transaction_date is
   a timestamp not a pure date

Write a query that returns: for each quarter of 2025, total
transactions, count, average size, and the first and last
transaction date in that quarter.

Drill 5 — CASE statements and categorization

CASE lets you create categories from raw numbers. This is how you
turn data into insight.

1. Write a query that categorizes each transaction into buckets:
   Small (<$100), Medium ($100-$1000), Large ($1000-$10000),
   Jumbo (>$10000). Count transactions per bucket.
2. Write a query that flags customers as Active (transactions in
   last 30 days), Dormant (last 30-90 days), At Risk (last 90-180
   days), or Lost (no transactions in 180+ days). Count customers
   per status.
3. Combine CASE with a window function: rank customers within
   their status category by total transaction amount.

Explain the difference between “searched CASE” (WHEN condition)
and “simple CASE” (CASE column WHEN value) and which one is more
common in analyst work.

Mini-Project — Period-Over-Period Treasury Report

Deliverable: an analyst-grade SQL report that answers a period-over-period question using window functions, date truncation, and CASE logic. Committed with a README that reads like a real analyst deliverable.

The question

Build a quarterly treasury performance report for 2025 showing, for each quarter:

  • Total transaction count and total transaction amount
  • Month-over-month change vs the previous quarter (absolute and percent)
  • Top 3 customer segments by contribution, ranked within the quarter
  • Categorical breakdown: what percentage of dollar volume came from Small / Medium / Large / Jumbo transactions
  • Month-over-month change in the categorical mix (is the business trending toward larger or smaller transactions?)

How to build it

Start a new Claude conversation in your Week 6 Project. Paste the requirements above. Build the query in CTE layers, following the iteration pattern. Expected sequence:

  1. CTE 1: base transaction rollup to quarter level
  2. CTE 2: add LAG columns for period-over-period
  3. CTE 3: segment ranking with RANK window function
  4. CTE 4: categorical breakdown with CASE
  5. Final SELECT: pull everything together

Each CTE you run separately to validate before layering the next. This is Skill #4 in action. Build, check, build, check.

Portfolio entry

Push to stephen-data-portfolio/week_06_quarterly_treasury_report/:

  • report.sql — the final query with CTE architecture, heavily commented
  • sample_output.csv — the actual results
  • README.md — written as an executive summary. Three bullets on what the report shows, five-line “so what” interpretation.

This piece is your SQL showcase. Recruiters click through to this when evaluating SQL skill.

How This Builds Forward

Phase 3 shifts gears. You stop writing SQL and start making things people look at. Dashboards. Charts. Visual storytelling. The SQL you just wrote is the data feeding those dashboards. You'll use the same schema in Looker Studio next week, but instead of writing queries you'll be designing visual answers to the same questions.

Halfway mark. Six weeks down, six to go. At this point you can hold your own in a technical interview on Excel and SQL. The second half teaches you to communicate what you find.
