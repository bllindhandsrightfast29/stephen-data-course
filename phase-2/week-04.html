Week 4 — SQL Basics

Phase 2 · SQL Skill Focus: Command AI to Execute (SQL Edition) Cert: Kaggle Intro to SQL ~12 hours
Course Progress
Week 4 of 12

SQL is the single most important technical skill for a data analyst. Every analyst job listing asks for it. Every interview tests it. This week you get comfortable enough to pass that test, using Claude as your pair partner and a browser-based SQL sandbox as your workspace. No installs, no server setup, no Anaconda.

Learning Goals

  • Read and write SQL with the four foundational clauses: SELECT, WHERE, GROUP BY, ORDER BY
  • Apply Skill #2 (command AI to execute) to a completely new tool and watch it transfer
  • Earn the Kaggle Intro to SQL certificate
  • Ship a SQL analysis of real transaction data to your portfolio

Why SQL Matters and Why DB Fiddle

SQL is the language databases speak. Companies store their operational data in databases. You pull data from those databases with SQL. If you can't write SQL, you can't be a data analyst. That's the whole pitch.

There are dozens of SQL tools. Most of them want you to install something. SSMS. MySQL Workbench. DBeaver. Docker. Anaconda. You're not going to install any of those. You're using DB Fiddle.

DB Fiddle is a browser-based SQL sandbox. No signup required. You paste a schema, paste some data, write queries, see results. It supports PostgreSQL, MySQL, SQLite. For learning SQL it's perfect. You'll use it all three weeks of Phase 2.

Alex the Analyst's SQL videos (if you watch them) use SQL Server Management Studio. The SQL itself is 95% identical. You translate his SSMS examples into DB Fiddle. Same language, different door.

The Week in Three Tracks

Track 1 — Kaggle Intro to SQL (the cert)

Go to Kaggle Learn Intro to SQL. Sign up for a Kaggle account (free). Work through the five lessons. At the end Kaggle gives you a certificate that lives in your Kaggle profile. You add it to LinkedIn. It counts.

Kaggle's platform runs SQL against Google BigQuery in the browser. You don't install anything. The queries you write there run against real datasets (Iraq news articles, OpenAQ air quality, Hacker News posts). Budget ~4 hours across Tuesday and Wednesday.

Track 2 — Claude SQL Drills in DB Fiddle

Five pair sessions with Claude, one per core clause. Each takes ~45 minutes. You'll practice against a treasury-flavored schema that Claude generates and you load into DB Fiddle.

Track 3 — The Mini-Project

A real SQL analysis using five queries against a treasury-style transactions database. Built Friday and Saturday.

AI Co-Work Pattern — Skill #2 Applied to SQL

The drill

Describe the question you want answered in plain English. Claude writes the SQL. You read every clause before you run it. You paste into DB Fiddle. You check the result against what you expected. You ask Claude what edge cases to watch out for.

This is Skill #2 from Week 2, applied to a different tool. The pattern transfers. That's the whole point.

Set up your treasury schema in DB Fiddle

Open a new Claude conversation inside your Week 4 Project. First prompt:

Generate a realistic treasury-flavored SQL schema I can paste into
DB Fiddle (PostgreSQL 15) to practice SQL this week.

I want 4 tables:
- customers: customer_id (pk), customer_name, segment (SMB/Mid/Enterprise),
  signup_date, country
- accounts: account_id (pk), customer_id (fk), account_type (Checking/
  Savings/Credit), open_date, current_balance
- transactions: transaction_id (pk), account_id (fk), transaction_date,
  amount, transaction_type (Deposit/Withdrawal/Transfer/Fee), description
- payments: payment_id (pk), account_id (fk), due_date, paid_date,
  amount, status (Paid/Overdue/Pending)

Generate CREATE TABLE statements plus INSERT statements for:
- 25 customers across all three segments, mix of countries
- 50-75 accounts (some customers have multiple)
- ~500 transactions across 2024-2025 with realistic distributions
- ~100 payment records, mix of statuses

Output the whole thing as one SQL script I can paste into DB Fiddle.
Make the data realistic (no obvious test values, varied amounts).

Paste Claude's output into DB Fiddle's schema panel. Click Run. Now you have a real database to query against.

The Five Drills

Drill 1 — SELECT and FROM

I have the treasury schema loaded in DB Fiddle. Walk me through:

1. How do I see all columns and all rows of the customers table?
2. How do I see just customer_name and segment?
3. How do I limit output to 10 rows?
4. How do I rename a column in output (segment to customer_tier)?

For each one, give me the SQL, run it in your head, and tell me
what rows you'd expect to see. Then I'll run it in DB Fiddle and
compare.

Also: explain why SELECT * is considered bad practice in production
even though it's fine for learning.

Drill 2 — WHERE

Using the transactions table, write SQL for each of these. Explain
each WHERE clause syntax.

1. All transactions over $1000
2. All withdrawals in the last 90 days
3. All transactions where the description contains the word "Wire"
4. All transactions where amount is between $100 and $500
5. All transactions that do NOT have a description (NULL handling)
6. All transactions that are either Withdrawals or Fees (IN clause)

For each one tell me:
- The SQL
- What rows I should expect based on the data you generated
- One gotcha (NULL handling, date format, string quoting)

Drill 3 — GROUP BY and basic aggregates

Now I want to summarize data, not just filter it. Write SQL for:

1. Count of transactions by transaction_type
2. Total amount by transaction_type
3. Average transaction amount by customer segment
4. Count of accounts per customer (which customers have the most
   accounts?)
5. Total deposits minus total withdrawals per account (net flow)

For each one:
- Write the SQL
- Explain which columns must be in GROUP BY vs which can be in the
  aggregate function
- Tell me what happens if I forget to put a non-aggregated column
  in GROUP BY

Drill 4 — ORDER BY and LIMIT

Now rank and sort. Write SQL for:

1. Top 10 customers by total account balance (descending)
2. The 5 oldest accounts by open_date (ascending)
3. The 3 months with the most transactions (GROUP BY + ORDER BY
   + LIMIT together)
4. Customers sorted by segment (alphabetical) then by signup_date
   (newest first) within segment

Explain:
- How ORDER BY ASC vs DESC works
- Why SQL wants ORDER BY before LIMIT in the statement
- What happens if two rows tie on the ORDER BY column

Drill 5 — Putting it all together

Write a single SQL query that answers this business question:

"For Enterprise-segment customers only, what are the top 5 customers
by total withdrawal amount in 2025? Show customer_name, total
withdrawals, and count of withdrawal transactions."

This needs SELECT, FROM, JOIN (we haven't formally covered JOIN but
you'll need one), WHERE, GROUP BY, ORDER BY, and LIMIT together.

Walk me through each clause in the order SQL executes them
(FROM runs first, SELECT runs almost last). Then I'll run
it in DB Fiddle.

This drill introduces JOIN ahead of Week 5 on purpose. You'll see it, understand it, and be ready for the real treatment next week.

Mini-Project — Five-Query Treasury Analysis

Deliverable: a SQL file with five queries and a README explaining what each query answers and what the results mean.

The five questions

  1. Which customer segment has the highest average account balance? By how much over the next-highest segment?
  2. What was the busiest month for transactions in 2025? How does it compare to the average month?
  3. Which accounts have had a net negative flow (withdrawals exceed deposits) over the past year?
  4. What percentage of payments are overdue, by customer segment? Is there a pattern?
  5. Of the top 20 customers by total account balance, how many are also in the top 20 by transaction count? (This one takes thinking. Two separate top-20 lists, then find the overlap.)

How to build it with Claude

Start a new conversation in your Week 4 Project called “Mini-Project Queries.” For each question:

  1. Write the question in plain English as a Claude prompt
  2. Claude writes the SQL, explains each clause
  3. You read the SQL before you run it. If a clause doesn't make sense, ask Claude to explain it deeper
  4. Paste into DB Fiddle, run, check the result
  5. Ask Claude: “What's a gotcha in this query? What assumption am I making?”
  6. Save the query with a comment header explaining what it does

Portfolio entry

Push to stephen-data-portfolio/week_04_sql_treasury_analysis/:

  • schema.sql — the CREATE TABLE and INSERT statements Claude generated
  • queries.sql — your five queries with comment headers
  • README.md — one paragraph on what the analysis shows, plus the business insight from each query

The README is the piece recruiters read. Make it short and tell a story. What did the numbers say about the business?

How This Builds Forward

Week 5 is all about JOINs. Every non-trivial SQL question in the real world pulls from two or more tables. This week's drills touched JOIN once (Drill 5) to plant the seed. Next week you learn INNER, LEFT, RIGHT, FULL, and when to use each. Same schema carries forward, so everything you load this week is still there Monday.
