Week 5 — Joins & Aggregates

Phase 2 · SQL Skill Focus: Validate Output (SQL Edition) ~11 hours
Course Progress
Week 5 of 12

Real SQL questions pull from two or more tables. This week you learn the joins: INNER, LEFT, RIGHT, FULL OUTER. Plus subqueries and CTEs. And you drill Skill #3 in SQL form: catching when a query returns the wrong number of rows because a join silently duplicated or dropped data.

Learning Goals

  • Read and write all four join types: INNER, LEFT, RIGHT, FULL OUTER
  • Use subqueries and CTEs to build multi-step queries readably
  • Apply Skill #3 (validate output against domain knowledge) to SQL results
  • Ship a “Customer 360” SQL analysis that pulls from every table in the schema

Why Joins Are the Scary Part

Almost every interview SQL question hinges on joins. They're scary because two mistakes look the same as correct answers:

  1. Duplicate rows — you joined to a one-to-many relationship and didn't realize it. Your total revenue comes back 3x what it should be because every customer row was duplicated three times.
  2. Missing rows — you used INNER JOIN when you needed LEFT. You're missing every customer who doesn't have a transaction yet. Your top-10 list is wrong because five customers got silently dropped.

Both mistakes return query results that look reasonable. The validation habit from Week 3 is what catches them. You know roughly how many customers you should have. If the query returns a weird number, something is off.

AI Co-Work Pattern — Skill #3 in SQL Form

The join validation drill

Every time Claude writes a query with a JOIN, run these checks before you use the result:

  1. Row count check — before running the full query, ask Claude to return just COUNT(*). Is the row count what you expected? If you had 25 customers and joined to transactions, the count should match something logical (number of transactions, or number of customers with transactions, depending on query intent).
  2. Grain check — what is one row in your result? One customer? One transaction? One customer-month? If you can't answer that in one sentence, the query isn't right.
  3. Null check — in a LEFT JOIN, rows from the left table with no match show NULLs for the right table's columns. Is that handled? Are you aggregating NULL-including columns correctly?

Example of the drill

I asked Claude to write a query:
"Show me total transaction amount per customer."

Claude wrote:
SELECT c.customer_name, SUM(t.amount) AS total_amount
FROM customers c
INNER JOIN accounts a ON a.customer_id = c.customer_id
INNER JOIN transactions t ON t.account_id = a.account_id
GROUP BY c.customer_name;

My row count check: result had 23 rows. I have 25 customers. Where
are the other 2?

I push back:
"I got 23 rows but I have 25 customers. Are two customers missing
because they don't have accounts or transactions? How do I get a
query that shows all 25 customers including zeros?"

Claude rewrites with LEFT JOIN. Now I get 25 rows, two showing $0.
That's the right answer for the question I actually meant to ask.

The Five Drills

Same treasury schema from Week 4 (reload it in a fresh DB Fiddle session if needed).

Drill 1 — INNER JOIN

Walk me through INNER JOIN. Specifically:

1. What does INNER JOIN return vs what does it exclude?
2. Write a query that shows customer_name alongside each of their
   accounts' account_type and current_balance
3. What happens if a customer has no accounts? What happens if an
   account has no customer (orphan)?

After you write the query, run a COUNT(*) version first so I can
verify the row count before looking at the data.

Drill 2 — LEFT JOIN

Now LEFT JOIN. Specifically:

1. Explain the difference between INNER and LEFT JOIN with a
   concrete example
2. Write a query that shows every customer, with their count of
   transactions (zero included for customers with none)
3. Write a query that finds customers who have an account but
   zero transactions in the past 90 days (classic "dormant"
   question)

For #3, walk me through why LEFT JOIN + WHERE clause on the right
table has a gotcha, and how to use WHERE vs ON-clause filtering
to handle it correctly.

Drill 3 — FULL OUTER JOIN and RIGHT JOIN

FULL OUTER JOIN and RIGHT JOIN are less common. Explain:

1. When would you use FULL OUTER? (Hint: reconciling two data
   sources that might each have records the other doesn't)
2. Why is RIGHT JOIN rare in practice? Could you always rewrite
   it as a LEFT JOIN?
3. Give me a concrete treasury example where FULL OUTER is the
   right answer. Something like reconciling our transactions
   table against a bank statement import.

Write the SQL and explain what rows appear and what NULLs mean in
each case.

Drill 4 — Subqueries

Now subqueries. Explain three patterns:

1. Subquery in WHERE clause (find customers whose total transactions
   are above average)
2. Subquery in FROM clause (summarize first, then query the summary)
3. Subquery in SELECT clause (the scalar one — for each customer,
   what percentage of all transactions is theirs?)

Write one example of each. Run a simpler non-subquery version first
for comparison so I see why the subquery is needed.

Also: explain when a CTE (next drill) is a better choice than a
subquery.

Drill 5 — Common Table Expressions (CTEs)

CTEs are subqueries that get a name and go at the top. Explain:

1. Syntax: WITH cte_name AS (...)
2. Why readability is the main reason to prefer CTEs over deeply
   nested subqueries
3. Multiple CTEs chained together (this is what senior analysts
   do)

Build one complex query as a single hard-to-read subquery chain,
then rewrite it as a CTE-based query. Show both. Explain which
one you'd want to see on a job-interview whiteboard.

The question: "For each customer segment, what's the average
transaction amount among customers who have at least 3 accounts?"

Mini-Project — Customer 360 Analysis

Deliverable: a single SQL query (as a CTE chain) that builds a complete “Customer 360” view. Plus a README with business interpretation.

The Customer 360 view should include, per customer

  1. Customer name and segment
  2. Total account balance (across all accounts)
  3. Count of accounts
  4. Total transaction count, last 12 months
  5. Total deposits and total withdrawals, last 12 months
  6. Net flow (deposits minus withdrawals)
  7. Count of overdue payments
  8. A computed “health score” from 1-5 based on the fields above (your logic, documented in comments)

How to build it with Claude

Start a new conversation in your Week 5 Project. Prompt:

I'm building a Customer 360 view using CTEs. I want one row per
customer with all the fields listed [paste the list above].

Help me build it step by step:
1. First, draft the CTE chain architecture: what CTEs do I need,
   and in what order?
2. Then we'll build each CTE one at a time. After each, I'll run
   it in DB Fiddle and validate the row count and sample values.
3. Finally we'll join all the CTEs together in the final SELECT.

Don't write the whole thing at once. Build it layer by layer so I
catch errors before they compound.

This is the iteration pattern from Skill #4 (next week). You're already using it.
